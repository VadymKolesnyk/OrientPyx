# OrientPyx — Задачі

## 1. Типи змагань (Discipline)

**Контекст:** зараз `EventDay.Discipline` — вільний рядок. Треба замінити на структурований тип.

**Вимоги:**
- Ввести перелік (enum або lookup-таблиця) типів змагань. На старті два значення:
  - **Заданий напрямок** — класика, учасник проходить всі КП у заданому порядку.
  - **За вибором по кількості КП** — учасник обирає будь-які КП, результат визначається за максимальною кількістю пробитих КП (або мінімальним часом при однаковій кількості).
- Архітектура має дозволяти додати нові типи пізніше (рогейн, естафета тощо) без переробки решти системи.
- На рівні дня (`EventDay`) зберігається **дефолтний тип змагань** для всіх груп цього дня.
- У UI (сторінка дня) — дропдаун замість текстового поля для вибору типу.

---

## 2. Групи (Groups)

**Контекст:** групи (наприклад М21, Ж21, М14, Відкрита тощо) є сутністю всього змагання, а не конкретного дня.

**Вимоги:**
- `Group` — сутність на рівні змагання (`CompetitionInfo`), зберігається в event.db.
  - `Id` (Guid)
  - `Name` (string) — назва групи, напр. "М21", "Ж14", "Відкрита"
  - `CreatedAt`
- Таблиця груп доступна зі сторінки/секції налаштувань змагання (CRUD: додати, перейменувати, видалити).
- Видалення групи блокується або попереджається, якщо для неї є дані на будь-який день.

---

## 3. Налаштування групи на день (GroupDaySettings)

**Контекст:** кожна група на кожен день має свої параметри дистанції.

**Вимоги:**
- `GroupDaySettings` — сутність, що прив'язана до `EventDay` + `Group`.
  - `Id` (Guid)
  - `EventDayId` (Guid, FK → EventDay)
  - `GroupId` (Guid, FK → Group)
  - `CourseOrder` (string) — порядок проходження КП у вигляді рядка (наприклад `"S1 31 32 33 34 35 F"`). Парситься пізніше.
  - `DistanceMeters` (int?) — довжина дистанції в метрах, необов'язково.
  - `DisciplineOverride` (nullable enum/lookup) — якщо заданий, перевизначає дефолтний тип змагань дня для цієї групи. Якщо null — використовується `EventDay.DefaultDiscipline`.
- UI: таблиця груп на сторінці дня, рядок на групу, колонки: назва, порядок КП, довжина, тип (із зазначенням чи це override чи дефолт).

---

## 4. Довідник контрольних пунктів (ControlPoints)

**Контекст:** на кожен день є свій набір КП, розставлених на місцевості.

**Вимоги:**
- `ControlPoint` — сутність, прив'язана до `EventDay`.
  - `Id` (Guid)
  - `EventDayId` (Guid, FK → EventDay)
  - `Code` (int або string) — номер/код КП, напр. `31`, `32`, або `"S1"`.
  - `Latitude` (double?) — широта (WGS-84).
  - `Longitude` (double?) — довгота (WGS-84).
  - `Type` (enum): `Regular` | `Start` | `Finish` — тип КП.
    - `Start` — старт (може бути декілька, напр. стартові коридори).
    - `Finish` — фініш.
    - `Regular` — звичайний КП.
- UI: таблиця КП на сторінці дня (або окрема вкладка). Колонки: код, тип, широта, довгота. CRUD.
- Координати необов'язкові — КП може існувати без GPS-координат.

## Залежності між задачами

```
Задача 1 (Discipline enum)  ← потрібна для Задачі 3 (override)
Задача 2 (Groups)           ← потрібна для Задачі 3 (GroupDaySettings)
Задача 3 (GroupDaySettings) ← залежить від 1 і 2
Задача 4 (ControlPoints)    ← незалежна, можна робити паралельно
```

Рекомендований порядок реалізації: **1 → 2 → 3**, **4** — паралельно або після.
